# T-01: Thread Primitives

> **Sample**: `27_BasicThread` | **Status**: Planned | **Depends On**: C-01 (Closures)

## 1. Executive Summary

This PRD introduces OS threading primitives — the foundation for true parallelism. Threads enable concurrent execution on multiple CPU cores.

**Key Insight**: Threads use OS APIs (pthreads on Linux, Windows threads on Windows). The raw pthread functions are accessed via Farscape-generated `Fidelity.Pthread` bindings. The language-level `Thread.create`/`Thread.join` API is a **CCS intrinsic** that wraps the raw bindings with closure handling and result slot management.

**Architecture**: Two layers:
- **Raw binding** (L1): `Fidelity.Pthread` generated by Farscape — `[<FidelityExtern("pthread", "pthread_create")>]`
- **Language API** (CCS intrinsic): `Thread.create` — handles closure→entry wrapper conversion, result slot allocation, ThreadHandle construction

The CCS intrinsic is justified because thread creation involves **compiler-managed state**: the thread entry wrapper must understand closure layout, and the result slot requires type-specific sizing. This is compiler work, not library work.

## 2. Language Feature Specification

### 2.1 Thread Creation

```clef
let threadFn () =
    Console.writeln "Hello from thread!"

let thread = Thread.create threadFn
```

### 2.2 Thread Join

```clef
Thread.join thread  // Wait for thread to complete
```

### 2.3 Thread with Closure

```clef
let counter = ref 0
let increment () =
    counter := !counter + 1

let t1 = Thread.create increment
let t2 = Thread.create increment
Thread.join t1
Thread.join t2
// counter may be 2 (or less due to race - see T-02 for synchronization)
```

### 2.4 Thread Return Value

```clef
let compute () = 42

let thread = Thread.create compute
let result = Thread.join thread  // Returns 42
```

### 3.2 Thread Intrinsics

`Thread.create`/`join`/`detach` remain CCS intrinsics because they require compiler knowledge of closure layout and result types:

```fsharp
// In CheckExpressions.fs
| "Thread.create" ->
    // (unit -> 'T) -> Thread<'T>
    let tVar = freshTypeVar ()
    NativeType.TFun(
        NativeType.TFun(env.Globals.UnitType, tVar),
        NativeType.TThread(tVar))

| "Thread.join" ->
    // Thread<'T> -> 'T
    let tVar = freshTypeVar ()
    NativeType.TFun(NativeType.TThread(tVar), tVar)

| "Thread.detach" ->
    // Thread<'T> -> unit
    let tVar = freshTypeVar ()
    NativeType.TFun(NativeType.TThread(tVar), env.Globals.UnitType)
```

### 3.3 Thread State Tracking

CCS may track thread creation for escape analysis — data passed to threads must remain valid.

## 5. Composer/Alex Layer Implementation

### 5.1 Thread Handle Structure

```fsharp
type ThreadHandle<'T> = {
    OsHandle: int64          // pthread_t or HANDLE
    ResultSlot: nativeptr<'T>  // Where result will be stored
    Closure: ClosureStruct     // The function to run
}
```

### 5.2 Thread Entry Wrapper

pthreads expects `void* (*)(void*)`. Alex generates a wrapper:

```fsharp
let threadEntryWrapper (arg: nativeptr<byte>) : nativeptr<byte> =
    let handle = NativePtr.read<ThreadHandle> arg
    let result = handle.Closure.CodePtr(handle.Closure.EnvPtr)
    NativePtr.write handle.ResultSlot result
    NativePtr.zero
```

### 5.3 Thread.create Witness

The witness calls into `Fidelity.Pthread.Thread.pthread_create` via ExternCall:

```fsharp
// Thread.create generates:
// 1. Allocate ThreadHandle (stack or region)
// 2. Store closure into handle
// 3. Allocate result slot
// 4. Call pthread_create (via ExternCall to Fidelity.Pthread)
// 5. Return handle as Thread<'T>
```

The `pthread_create` call itself goes through the standard ExternCall pathway — same `pExternCallResolved` witness as GTK functions (D-01).

### 5.4 Thread.join Witness

```fsharp
// Thread.join generates:
// 1. Extract OS handle from ThreadHandle
// 2. Call pthread_join (via ExternCall to Fidelity.Pthread)
// 3. Read result from result slot
// 4. Return typed result
```

## 6. MLIR Output Specification

### 6.1 Thread Handle Type

```mlir
// ThreadHandle is memref<Nxi8> (flat byte buffer):
//   offset 0: os_handle (i64 — pthread_t)
//   offset 8: result_slot (index — pointer to result memref)
//   offset 16: closure (closure bytes — code_ptr + captures)
```

### 6.2 Thread Entry Wrapper

```mlir
func.func @thread_entry(%arg: index) -> index {
    // ... extract closure, call, store result ...
    %ret_zero = arith.constant 0 : index
    func.return %ret_zero : index
}
```

### 6.3 Thread Creation

```mlir
// pthread_create uses ExternCall pathway
func.func private @pthread_create(index, index, index, index) -> i32
    attributes { "link" = "pthread" }

// let thread = Thread.create myFunc
%handle = memref.alloca() : memref<Nxi8>
// ... store closure, allocate result slot ...
%handle_idx = memref.extract_aligned_pointer_as_index %handle : memref<Nxi8> -> index
%attr_zero = arith.constant 0 : index
func.call @pthread_create(%os_handle_ref, %attr_zero, @thread_entry, %handle_idx)
    : (index, index, index, index) -> i32
```

let work (id: int) () =
    Console.write "Thread "
    Console.write (Format.int id)

        sum <- sum + i

    Console.writeln "=== Basic Thread Test ==="

    let t1 = Thread.create (work 1)

## 9. Design Decision: Why Thread.create is an Intrinsic

`Thread.create` could theoretically be a library function, but it requires **compiler cooperation**:

1. **Closure layout knowledge**: The thread entry wrapper must extract the code pointer and environment from the closure. This layout is compiler-determined.
2. **Result type sizing**: The result slot size depends on the return type of the closure. Only the compiler knows this at monomorphization time.
3. **Entry wrapper generation**: The `@thread_entry` function is compiler-generated per instantiation.

The raw `pthread_create` is a library function (ExternCall via Farscape). The `Thread.create` wrapper around it is a compiler intrinsic. This is analogous to how `Console.writeln` is an intrinsic that internally calls `write` (which will become an ExternCall via `Fidelity.Libc`).
