# WASM Deployment and Lifecycle on Cloudflare

**SpeakEZ Technologies | Fidelity Framework**
**April 2026**

Containers and Workers are clear mental models. A Container is a standalone Linux process with its own lifecycle, filesystem, and addressable endpoints. A Worker is a JavaScript-shaped request handler that runs in a V8 isolate with Cloudflare-specific runtime bindings. WASM on Cloudflare sits alongside both but is not analogous to either, and the mental model takes a moment to lock in. This document establishes the model.

The one-sentence version: **WASM on Cloudflare is a library, not a service.** It is not a deployable unit of its own. It ships embedded inside a Worker's deployment bundle, always. The Worker (which is always a JavaScript module at its entry point, regardless of what language you wrote) is the outer shell; the WASM is inner compute that the shell calls into.

## The Containment Hierarchy

From outermost to innermost, a Cloudflare WASM deployment has five layers:

1. **Deployment artifact.** The bundle `wrangler deploy` uploads. Contains JavaScript module(s), WASM modules, metadata, and configuration.
2. **Worker.** Cloudflare's deployable unit. Always has a JavaScript entry point (`export default { async fetch(request, env, ctx) { ... } }`). Holds references to the WASM modules bundled with it.
3. **JavaScript shim.** Either hand-written (if the author wrote JavaScript that happens to call into WASM) or auto-generated (if the author wrote Rust via `workers-rs`, in which case `worker-build` generated the shim). Its responsibilities: receive the request, access the `env` bindings, marshal JavaScript values into WASM value types, call WASM exports, marshal the results back, and return a response.
4. **WASM module (`WebAssembly.Module`).** The compiled WASM code, produced from the `.wasm` file. Cloudflare caches this at the edge to amortize compile cost; multiple isolates at the same edge location can share a single compiled `Module`.
5. **WASM instance (`WebAssembly.Instance`).** The instantiated module with its own linear memory and its imports bound to specific JavaScript functions. Created per isolate during the Worker's top-level script execution.

Each layer exists for a reason. The deployment artifact is what Cloudflare's distribution network moves; the Worker is the isolation boundary and the unit of billing; the JS shim is the bridge between Cloudflare's JavaScript-shaped runtime API and the WASM module's typed function surface; the `Module` is cacheable compile output; the `Instance` is per-isolate state.

## What a Deployment Bundle Contains

A minimal Cloudflare Worker deployment that uses WASM contains:

```
<worker-name>/
├── worker.js             # The JS entry point (hand-written or generated)
├── module.wasm           # The compiled WASM module
└── wrangler.toml         # Configuration: entry point, bindings, routes
```

For a `workers-rs`-produced Rust Worker, the bundle is conceptually the same, but the `worker.js` file was generated by `worker-build` rather than written by hand:

```
<worker-name>/
├── build/
│   ├── worker/
│   │   ├── index.js      # Auto-generated JS shim
│   │   └── module.wasm   # Rust compiled to WASM via wasm-bindgen
│   └── ...
└── wrangler.toml
```

The generated `index.js` contains the import of the `.wasm` module, the instantiation logic, the wasm-bindgen glue code that unpacks WASM values for JavaScript consumption and vice versa, and the `fetch` handler that routes into the Rust-exported entry function.

There is no deployment mode in which WASM ships without a JavaScript entry point. The Workers runtime's entry points (`fetch`, `scheduled`, `email`, `queue`, `tail`) are all JavaScript-facing APIs. A request arriving at a Cloudflare edge location hits JavaScript first, always.

## The Cold Start Sequence

When a request reaches an edge location and there is no warm isolate for the target Worker:

1. Cloudflare loads the Worker's JavaScript entry point module. The top-level code runs once.
2. Top-level code includes `import wasmModule from "./module.wasm"` (modern module syntax). Cloudflare's runtime intercepts this to return a `WebAssembly.Module` — the compiled form, drawn from the edge location's module cache if available, compiled from bytes if not.
3. Top-level code also includes `new WebAssembly.Instance(wasmModule, imports)` (or the wasm-bindgen equivalent). This creates the linear memory, allocates any tables, resolves imports to JavaScript functions, and produces an `Instance`.
4. The `fetch` handler function is now defined, with the WASM instance in closure scope or module-level variable reach.
5. The incoming request invokes the `fetch` handler. The handler reads from `env`, marshals request data as needed, calls into WASM exports, marshals results, constructs a `Response`, and returns.

Subsequent requests on the same warm isolate skip steps 1-4 entirely. The instance is already there. The `fetch` handler runs against live state.

Cold-start cost, in practice:

- Step 1 (JavaScript parse/eval): typically sub-millisecond for small shims, a few milliseconds for large ones.
- Step 2 (WASM module compile): this is the expensive step when it happens, and what Cloudflare caches aggressively. For a ~1 MB `.wasm` file, uncached compile can take 50-200 ms; cached lookups take sub-millisecond.
- Step 3 (instantiation): fast once the module is compiled — typically single-digit milliseconds.
- Step 4-5: normal request handling.

The total cold-start overhead is typically 5-50 ms for reasonable WASM sizes, most of it WASM compile if not cached. Warm requests have no WASM-attributable cost beyond the call-into-WASM and marshal-back trips.

## Why the JavaScript Shim Is Unavoidable

A natural question: if 99% of my Worker is Rust, why do I still need any JavaScript? The answer is structural, not just convention:

- **Cloudflare's runtime API is JavaScript-shaped.** `fetch` handlers receive a `Request` object, return a `Response` object, and operate on `env` bindings. These are JavaScript-defined interfaces. The WASM module has no native access to them; WASM's value types are `i32`, `i64`, `f32`, `f64`, `v128`, `funcref`, and `externref`. A `Request` is a JavaScript object, not a WASM value.
- **Bindings injection happens into the JavaScript scope.** When Cloudflare's runtime spins up your Worker's isolate, it injects `env.MY_KV`, `env.R2_BUCKET`, `env.MY_QUEUE` into the JavaScript context. The WASM module cannot see `env` directly. The JS shim reads `env`, exposes specific operations to WASM via imports (as functions the WASM module can call), and routes the results back.
- **Async semantics need Promise integration.** Cloudflare bindings return Promises (`env.MY_KV.get(key)` returns `Promise<string | null>`). The WASM module can't directly await a Promise in pre-JSPI runtimes (and even with JSPI, the binding has to be authored to cooperate). The JS shim handles the Promise lifecycle and either polls the WASM module or uses JSPI to suspend the WASM call until the Promise resolves.

For `workers-rs` Rust Workers, the JS shim is auto-generated and typically ~100 lines. You don't see it in your source tree until you peek at the build output. But it is there, because it has to be.

## Cross-Worker Composition

WASM cannot be invoked across Workers directly. There is no "WASM endpoint" primitive in Cloudflare's architecture. Composition between Workers happens at the Worker level, and any WASM used by Worker B is encapsulated inside B's deployment.

**Service bindings** are the HTTP-style RPC mechanism between Workers:

```
Worker A's wrangler.toml:
  services = [
    { binding = "BACKEND", service = "worker-b" }
  ]

Worker A's code:
  const response = await env.BACKEND.fetch(request);
```

From Worker A's perspective, `env.BACKEND` is a service it calls. From Worker B's perspective, it's just a request arriving at its fetch handler — identical in shape to an HTTP request from outside. Worker B may invoke its WASM internally; that's opaque to Worker A.

**Durable Objects with WASM** are another composition pattern:

```
Worker A's code:
  const doStub = env.COUNTER_DO.get(id);
  const response = await doStub.fetch(request);
```

The DO class (defined in JavaScript, possibly with WASM embedded in its own deployment) receives the request and handles it. Each DO instance has its own lifecycle; its WASM instance — if any — is instantiated when the DO activates and persists across requests to that DO until it evicts. The DO is addressable by ID, the WASM inside is not.

There is no pattern for "Worker A calls Worker B's specific WASM function." All communication between Workers is HTTP-request-shaped, and all WASM is encapsulated inside the Worker that bundled it.

## Clean Tier Contrast

The operational differences in one table:

| Aspect | Container | Worker (with or without WASM) | WASM Module |
|:-------|:----------|:------------------------------|:------------|
| Deployable unit | Yes (container image) | Yes (script + assets) | No (bundled inside a Worker) |
| Lifecycle | Long-running process | Isolate-per-request (reused when warm) | Instantiated per isolate with its host Worker |
| Addressable endpoint | Direct (DNS, IP, container URL) | URL route or service binding | Not addressable independently |
| Filesystem | Yes | No | No |
| Network access | Arbitrary sockets | Cloudflare-mediated `fetch` | None directly; goes through host JS |
| Runtime API | POSIX | Workers runtime APIs (fetch, bindings) | None directly; accessed via JS imports |
| Can hold state across requests | Yes (process memory, disk) | Only via Durable Objects or external stores | Yes, inside its host Worker's isolate (same as Worker) |
| Scaling unit | Per-container | Per-request (isolates ephemeral) | Scales with host Worker |
| Independent deployment | Yes | Yes | No |
| Observability | Cloudflare container metrics + logs | Cloudflare Worker metrics + logs | Subsumed into host Worker's metrics |

The "WASM Module" column's repeated "no" entries (not a deployable unit, not addressable, no filesystem, no direct network, no independent deployment, no separate observability) are the answer to "how is a WASM instance distinct?" It's not a separate primitive at all. It's compute inside a Worker.

## The Useful Analogy

The analogy that captures this best: a WASM module on Cloudflare is the equivalent of a statically-linked `.so` or `.dll` in a traditional executable. You ship the library alongside the executable; at program start, the library is loaded and linked; the executable calls the library's exported functions; the library has no existence or invocation model separate from the executable loading it.

The analogies that do *not* work:

- **Microservice.** Microservices have independent deployments, independent scaling, independent endpoints, and independent lifecycles. WASM modules have none of these things.
- **Container.** Containers are long-running processes with filesystems and addressable endpoints. WASM modules are inert code bytes that only exist as instances inside a Worker's isolate.
- **Service binding target.** Service binding targets are Workers. WASM is not an addressable endpoint; it's what a Worker happens to contain.
- **Shared library loaded by multiple Workers.** Different Workers that want to use the same WASM module each bundle their own copy. There is no cross-Worker WASM reuse at runtime.

## Implications for Fidelity WASM Deployments

**Every Fidelity Worker that uses WASM is, at deployment time, a bundle of JavaScript shim + WASM module(s).** Composer's backend produces both: the WASM comes from the LLVM WASM or WAMI pathway; the JavaScript shim is either hand-equivalent code (if emitting everything from Clef) or a thin wrapper (if Fidelity follows the `workers-rs` `worker-build` pattern of auto-generating the shim). Either way, the deployment artifact is always a combined JS+WASM bundle.

**A Fidelity actor that runs as a WASM-backed Cloudflare Worker is structurally the same shape as the pure-JS actors documented in Fidelity.CloudEdge.** The actor's external interface is the Worker's fetch handler. Its internal compute is WASM. Other Workers, other DOs, and external clients interact with it via HTTP (or WebSocket, or MoQ stream); none of them see the WASM directly. BAREWire-framed messages over these transports are processed by the WASM codec inside the Worker's isolate.

**Composer's backend decision of "should this code compile to Tier 1 (pure JS Worker), Tier 2 (WASM-in-Worker), or Tier 3 (Container)" is a deployment-shape decision as much as a performance decision.** Tier 1 and Tier 2 produce the same deployment primitive (a Cloudflare Worker); Tier 3 produces a Cloudflare Container, which is fundamentally different operationally. The Tier 1 vs. Tier 2 choice shifts compute into WASM inside the Worker; the Tier 2 vs. Tier 3 choice shifts compute into a separate long-running process with its own identity. These are different magnitude of architectural move, and they should be distinguished clearly in target-profile logic.

**The encapsulation-by-Worker model simplifies reasoning about cross-Worker trust boundaries.** A Fidelity application composed of multiple Workers (e.g., a public-facing Worker doing HTTPS termination and auth, a backend Worker with BAREWire codec in WASM, a Durable Object for stateful coordination) has each Worker as its own isolation boundary. WASM inside any of them cannot escape the host Worker's isolate. Supply chain considerations from [javascript-targeting/05_supply_chain_and_transcribe.md](../javascript-targeting/05_supply_chain_and_transcribe.md) apply per Worker: each bundle is its own security perimeter.
